# NetLogic C# API and tooling notes

Source basis:

- Rockwell Knowledgebase PDF: `FactoryTalk Optix Frequently Asked Questions - Programming - C# and Netlogic`
- Rockwell Knowledgebase PDF: `Error in FactoryTalk Optix after installing Visual Studio Code and the C# SDK`

These notes are sanitized summaries. Do not commit downloaded Knowledgebase PDFs unless redistribution is clearly allowed.

## Key conclusion

For serious FT Optix project automation, the best current path is probably:

```text
DesignTime NetLogic C# API  -> primary project-generation path
NodeSet XML import/export   -> verification and regression-test path
YAML project files          -> pattern research and diff/reference path
```

XML import/export remains useful and has already worked for Model-folder probes. However, FT Optix C# NetLogic is a better candidate for generating native FT Optix nodes because it runs inside the Studio/project environment and can use FT Optix APIs directly.

## What the Programming FAQ confirms

The Programming - C# and NetLogic FAQ confirms:

1. NetLogic examples are available in the public FactoryTalk Optix GitHub repository.
2. Copy/paste C# snippets are available in the public FactoryTalk Optix cheat sheet.
3. FactoryTalk Optix C# supports `.NET 6` bindings.
4. Additional packages can be added through NuGet Package Manager.
5. Examples exist for manipulating variable values with C#.
6. Installed NetLogic can be found from FactoryTalk Optix Studio's Project View search field.

Important implication:

```text
FT Optix does support meaningful C# API / DLL / NuGet-based extension work,
but this is still NetLogic-centric rather than a known Studio 5000 LDSDK-style
external project SDK.
```

## What the VS Code / C# SDK error PDF confirms

The VS Code / C# SDK error PDF documents problems after installing Visual Studio Code and the C# SDK.

Observed error categories:

```text
NU1301: Unable to load the service index for source https://api.nuget.org/v3/index.json
NU1100: Unable to resolve Microsoft.NETCore.App.Ref (= 6.0.25) for net6.0
NU1100: Unable to resolve Microsoft.WindowsDesktop.App.Ref (= 6.0.25) for net6.0
NU1100: Unable to resolve Microsoft.AspNetCore.App.Ref (= 6.0.25) for net6.0
```

Suggested fixes from the PDF:

### Solution 1 - clear VS Code cache

1. Close Visual Studio Code.
2. Open Task Manager and end remaining Visual Studio Code background processes.
3. Open:

```text
C:\Users\<user_name>\AppData\Roaming\Code
```

4. Clear the contents of these folders:

```text
Cache
CachedData
CachedExtensions
CachedExtensionVSIXs  (if present)
Code Cache
```

### Solution 2 - delete NuGet.config

Delete the NuGet config file under:

```text
C:\Users\<username>\AppData\Roaming\NuGet\NuGet.config
```

## Practical setup notes

For a PC that will be used for FT Optix NetLogic development, install and verify:

```text
FactoryTalk Optix Studio
Visual Studio Code
C# extension
.NET Install Tool
C# Snippets
NuGet Package Manager GUI
Internet/proxy access to NuGet if packages must be restored
```

If NetLogic editing or restore fails, check the VS Code cache and NuGet config before assuming the FT Optix project is broken.

## Recommended automation direction

Do not make raw YAML editing the main write path yet. YAML is very useful for learning FT Optix patterns from sample projects, but direct manual editing may be fragile across FT Optix versions.

Do not abandon XML either. It is already proven useful for import/export verification and for learning how FT Optix normalizes nodes.

Use DesignTime NetLogic as the next serious probe:

```text
Probe 04A - DesignTime NetLogic creates Model nodes
```

Target output:

```text
Model
└─ AI_NetLogicProbe_01
   ├─ StatusText
   ├─ TestNumber
   ├─ Running
   └─ Pump1
      ├─ SetSpeed
      └─ CurrentSpeed
```

Expected workflow:

```text
1. Create a DesignTime NetLogic in FT Optix Studio.
2. Add an exported method such as GenerateModelProbe().
3. Use C# API calls to create objects and variables under Model.
4. Run the method from Studio.
5. Save the project.
6. Export the generated node back to NodeSet XML.
7. Compare the exported XML against expected structure.
```
